# Mast Remote Substrate Spike

## Decision Target

Mast must replace the current Chorus/Zed workflow and become the engineer cockpit for SAIL:

```text
Mac Mast UI
  -> SAIL host / Incus project container
  -> remote IDE backend over SSH or equivalent tunnel
  -> Java indexing, refactoring, Git, Maven, tests, terminal, debugging
  -> ACP/agent threads running in the same remote workspace
  -> SAIL specs, kanban, snapshots, branches, dispatch, logs, and PR review
```

The spike is not about a remote dashboard. It is about project-scoped remote execution where code intelligence and agents share the same filesystem, Git state, environment, and trust boundary.

## What JetBrains Exposes Publicly

JetBrains Remote Development uses a local client/Gateway connected to a backend IDE running on a remote host. The backend owns source code, indexing, builds, tests, debugging, and plugins. This is the closest known substrate to the SAIL target.

Documented capabilities that matter for Mast:

- SSH connection from IntelliJ IDEA, Toolbox, or Gateway to a remote host.
- Backend IDE download or custom installer upload.
- Manual backend startup using `remote-dev-server.sh run <project>`.
- Gateway links that can be generated by the backend and opened locally.
- Backend plugin installation with `remote-dev-server.sh installPlugins <PROJECT_PATH> <PLUGIN_ID>`.
- Gateway customization for internal orchestration services, with APIs described as not final and custom-deal oriented.

## What The Current Mast Source Build Shows

The current source-built product is still the open-source `ideaIC`/`idea-ce` product line:

- `IdeaCommunityProperties` sets the base artifact name to `ideaIC-*`.
- Its bundled plugin list is `IDEA_BUNDLED_PLUGINS + intellij.javaFX.community`.
- `IDEA_BUNDLED_PLUGINS` includes Java, Maven, Gradle, Git, GitHub, terminal, MCP server, Kotlin, and related community plugins.
- It does not include `intellij.remoteDevServer`, `intellij.gateway`, or `intellij.gateway.ssh`.
- The build infrastructure only adds Remote Development libraries when bundled plugin modules contain `intellij.remoteDevServer`.
- Source references and DSL docs know about Gateway, but the actual Gateway product modules appear absent from the OSS checkout.

Initial conclusion: our `ideaIC` artifact is good enough to validate local Java IDE behavior, but it is not yet a Mast remote substrate.

## Relationship To Chorus/Zed

Chorus/Zed wires remote project support and agent support through the same project abstraction:

- The remote server owns worktree, buffer, LSP, Git, task, DAP, and agent server stores.
- The local client sees remote proxies for those stores.
- ACP agent commands are resolved through the remote project.
- When a project has a remote client, ACP stdio is wrapped through the remote connection, so the agent process runs remotely.

Mast needs the same property. A local SAIL panel is not enough.

## Architecture Options

### Option A: Use Official JetBrains Remote Development As The Substrate

SAIL provisions the host/container and generates a Gateway-compatible connection flow. The IDE backend runs in the SAIL environment. Mast/SAIL plugins are installed on the backend and client as required.

Expected flow:

```text
sail project open <project>
  -> ensure container running
  -> ensure SSH access / tunnel
  -> ensure IDE backend present
  -> install/update Mast SAIL backend plugin
  -> run remote-dev-server.sh run <workspace>
  -> open jetbrains-gateway://connect#...
```

Pros:

- Reuses JetBrains' remote file/index/debug/run/test infrastructure.
- Matches the thin-client model we need.
- Gives Java DX quickly if licensing/distribution constraints are workable.

Risks:

- Gateway/RemoteDev pieces may not be redistributable in Mast.
- Custom orchestration APIs may require JetBrains relationship or proprietary artifacts.
- Backend plugins are per remote project, so SAIL must manage plugin installation/version drift.

### Option B: Build Mast As A Gateway-Orchestrated Product

Try to add/bundle the missing Gateway/RemoteDev modules into the Mast source build.

Pros:

- Most branded and integrated path if possible.

Risks:

- OSS source does not appear to contain required Gateway implementation modules.
- Product layout references `UltimateModuleSets.ssh()` and `intellij.gateway.ssh` examples, but these modules are absent locally.
- Could become a licensing/product dead end.

### Option C: Build A SAIL-Specific Remote Layer

Write an IntelliJ plugin/backend pair that mimics the Zed remote project model for SAIL.

Pros:

- Fully OSS and fully controlled.
- Can be tailored exactly to SAIL/Incus/spec/agent workflows.

Risks:

- Very large effort: remote VFS, indexing, project model, Git, terminals, process execution, tests, debugger, file watchers, and agent stdio all become our problem.
- Hard to match JetBrains Remote Development polish quickly.

## Spike Plan

### Stage 1: Official JetBrains Remote Smoke

Use official IntelliJ IDEA/Gateway on a Mac and a real SAIL project container.

Validate:

- SSH connection to host/container.
- Open remote Java/Maven project.
- Indexing and navigation.
- Git status/diff.
- Terminal in remote workspace.
- Maven/test run.
- Backend control center behavior under SAIL CPU/memory limits.

### Stage 2: Manual Backend Launch

Install or upload an IDE backend in the SAIL environment and run:

```bash
bin/remote-dev-server.sh run /home/dev/workspace/<repo> \
  --ssh-link-host <host> \
  --ssh-link-user <user> \
  --ssh-link-port <port>
```

Validate:

- Generated `jetbrains-gateway://connect#...` link opens locally.
- Project reconnects after backend restart.
- Install path and cache paths are compatible with SAIL container lifecycle.

### Stage 3: Plugin Placement

Install a minimal SAIL/Mast plugin into the remote backend.

Validate:

- Plugin can call local/remote SAIL API safely.
- Plugin can see project path, Git root, and IDE project model.
- UI appears in JetBrains Client as expected.
- Backend vs client plugin placement is understood.

### Stage 4: Agent Path

Validate project-scoped ACP/agent execution.

Validate:

- Codex/Claude process runs inside the remote SAIL workspace.
- Agent cwd is the remote project root.
- Agent can read/write files through IDE/ACP or filesystem as intended.
- Spec context can be passed from SAIL into agent sessions.
- Logs and status can round-trip to the Mast UI.

## Security Requirements

- Do not expose SAIL API remotely without explicit opt-in and bearer-token auth.
- Keep API tokens owner-readable only and never place them in generated Gateway links.
- Treat spec markdown and agent output as untrusted input.
- Validate project names, repo paths, branch names, spec ids, and generated SSH/Gateway parameters before command execution.
- Avoid logging secrets in plugin, backend, Gateway link handling, or ACP debug output.
- Keep agent execution inside the project container, not the Mac.

## Initial Recommendation

Prioritize Option A for the spike. It is the only path likely to deliver world-class Java DX and remote agents quickly.

Do not invest in a custom remote layer until Option A is proven blocked by licensing, missing APIs, unacceptable UX, or inability to run agents in the same remote environment.

Keep the Mast source fork alive for branding, local experiments, agent workbench exploration, and OSS plugin development, but treat it as unproven for remote development until it can produce or orchestrate a backend with `remote-dev-server.sh`.
